Lab data, one front door — and a path everyone can adopt

This page summarizes where Tech for Cancer is headed with Uri Ben-David’s lab: the solution direction we are taking, what we are aiming for, and what we will start with first — including decisions from the latest working session with Haia (bioinformatics lead).

Companion to the full research deck: Research page (hosted)

Context
Fragmented data & workflows

Many storage locations, no catalog, heavy load on Haia for computation. Wet-lab work is individualized by design; we do not need to standardize every intermediate step.

Direction
A lab portal + shared context

One browser “home” for starting research, uploading outputs, searching, and guided steps — wired behind the scenes to agreed storage and tools.

First move
Computational pilot

Simplest flow with people most likely to adopt (50/50 and 100% computational students), then expand. Validate hosting & network with university IT.

High level

The solution we are going with

We are not asking everyone to abandon Excel, Slack, or personal folders overnight. The bet is a thin, friendly layer — a lab portal — that becomes the default place to start work, hand off outputs, and find what matters, while integrations push data into shared, backed-up locations without users juggling paths and servers.

1

Lab portal (single entry point)

A web experience researchers open in the browser — similar in spirit to setting the lab intranet or a team homepage as the start page. From there they can:

  • Start or continue a research thread (guided questions: topic, related prior work, uploads).
  • Drag-and-drop files so uploads land in the right shared place without hunting for folders.
  • Search across connected storage (e.g. lab drive / server paths we wire in).
  • Use checklists for required steps (e.g. backup, naming, QC) so compliance is visible, not tribal knowledge.
  • Optional: light inventory hooks (e.g. tagging reagent use) — “behind the scenes” decrements can be modeled later.
2

Shared context (markdown + skills mindset)

Meetings and decisions accumulate as structured context (e.g. markdown in a repo or folder the lab owns) so assistants and future automation know who owns what, which tools apply, and how analyses should be validated. This mirrors how we work internally: one place for truth, iterated over time — the lab does not have to use GitHub, but the idea is the same.

3

AI where it helps — not where it leaks

Skills / prompts can encode: “check for similar past work,” “produce tables with these statistical rules,” “push results back to the repo.” Enterprise vs shared Pro accounts and API usage must align with university policy and NDA constraints — validated before any patient or collaborator data flows through external APIs.

Researcher → Portal UI → Routing + checklist → Shared storage / server → Optional AI (policy-safe)

Wet lab vs dry lab (Haia, March 15): For wet-lab work, the pattern is similar across people even if filing differs — focus on start (“has anyone done this experiment?”) and end (outputs, backup, what enters a paper). The messy middle is often disposable; we should not force everyone’s mind-map or filing style to change. For computational work, unifying pipelines, GitHub, and server usage is the higher-leverage standardization.

Outcomes

What we aim for

Success is not “everyone uses one app for everything.” It is shorter time from question to trusted result, less duplicate effort, and knowledge that survives student turnover.

  • Reduce the bioinformatics bottleneck: repeatable flows so not every analysis queues through Haia for the same class of task.
  • One coherent picture of lab data: a path toward a catalog / index of what exists, where it lives, and who owns it.
  • Standardization where it compounds: shared environments, reference data, pipelines, and Git — especially for computational work.
  • Adoption by design: the weakest link is not the tech — it is whether the easiest user finds the portal obviously easier than the status quo. The experience must work for low tech-savvy users (guided clicks, not terminal literacy).
  • Safe AI: insights and QC assistance only where data classification and university policy allow; otherwise on-prem or deferred.
Scope

What we will start with

We start narrow: one persona track, the simplest end-to-end flow, and people most likely to partner and give feedback — not the hardest case first.

A

Computational group first

Agreement with Haia: the computational side is the best first beachhead — they already push toward shared server, GitHub, and unified pipelines; success is visible to others at the bench. Wet-lab–only workflows stay second unless a very simple “start / finish” pattern is enough.

B

Simplest flow + strongest adopter

Pick one flow (e.g. a defined pipeline or “new analysis” path) and one or two researchers who are motivated and sticky (enough time left in the lab, willing to try tools). Candidates discussed: 50/50 wet/dry students (e.g. Adi and a peer) and/or the newer 100% computational student — people who will stress-test the portal and Git patterns without disappearing next semester.

C

Validate IT & infra before big build

Before scaling UI: confirm where the portal can be hosted (university server vs external), VPN / GlobalProtect access to internal storage, connectors to lab file server and OneDrive/Microsoft APIs, and policy on external LLM APIs for research data. Haia can relay questions to campus IT; we proceed when we have “good to go” answers.

Parallel track (already in motion): Haia continues unifying pipelines, GitHub practice, and server usage — the portal should ride on that foundation, not replace it overnight.

March 15, 2026

Sync with Haia — what we aligned on

Working session (audio transcribed). Below are the commitments and principles that feed the sections above.

Portal definition

A browser home page for the lab: guided flows (e.g. new vs continuing research), uploads, search, optional AI — all routing to shared storage behind the scenes so users “just drag here” without learning folder topology.

Wet vs dry

Do not fight individual mind-maps for bench work. Standardize inputs/outputs and backups; leave intermediate chaos lower priority. Computational standardization has higher ROI first.

Security & access

Allowlist lab members; not an open internet wiki. University network (VPN) and hosting constraints must be checked before assuming remote access to servers works like a SaaS product.

Claude / AI

Shared non-enterprise accounts blur privacy between users; Pro vs Enterprise and API usage matter for team isolation and cost. Align with lab + university rules before sending sensitive data to any external model.

Pilot people

Prefer 50/50 researchers with passion for computation + runway (MD-PhD), and/or dedicated computational students — they give the best feedback loop for the first portal slice.

Next step

Haia to think through technical blockers; Tal to send a concise IT question list; iterate on WhatsApp; in-person lab visit when practical.

Dependencies

IT & infrastructure questions (before scale)

These need yes/no or owner from university IT — not blockers for a paper prototype, but blockers for production and data-bearing integrations.

Can a small web app be hosted on approved university infrastructure (VM, department web, or existing compute server with HTTP)?
IT / Haia
Is access from off-campus via GlobalProtect (or equivalent) sufficient for lab members to use an internal portal?
IT
What protocols/APIs are allowed to reach the lab file server and shared drives from an application (SMB, Graph API for OneDrive, etc.)?
IT
What is the policy on sending research or collaboration (NDA) data to external APIs (e.g. Claude/OpenAI)? Is anonymization enough?
Compliance / PI
Is SSO / university identity available for a lab tool, or is email allowlist acceptable for v1?
IT
Who funds API usage and hosting if not on free tier — lab grant, department, or T4C?
